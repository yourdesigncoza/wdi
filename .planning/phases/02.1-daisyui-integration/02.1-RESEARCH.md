# Phase 02.1: DaisyUI Integration - Research

**Researched:** 2026-02-06
**Domain:** DaisyUI v5 component library integration with Tailwind CSS v4
**Confidence:** HIGH

## Summary

This phase integrates DaisyUI v5 (the most popular Tailwind CSS component library) into the existing React 19 + Vite + Tailwind CSS v4 frontend. DaisyUI v5 is purpose-built for Tailwind CSS v4 and uses the new `@plugin` directive -- no config files needed. The project already has the correct Vite + Tailwind v4 setup (`@tailwindcss/vite` plugin, `@import "tailwindcss"` in CSS), so integration is minimal: install the package and add one line to `index.css`.

The migration involves converting existing hand-rolled Tailwind utility classes to DaisyUI semantic component classes (`btn`, `modal`, `card`, `alert`, `input`, `select`, `textarea`, `navbar`, `link`, `loading`). All existing Tailwind utility classes continue to work alongside DaisyUI classes. Theme configuration uses built-in themes (`corporate` for light, `business` for dark) which provide a professional, trustworthy aesthetic suitable for a South African legal will platform.

DaisyUI's component classes dramatically reduce code verbosity (e.g., a button goes from `px-6 py-2.5 bg-blue-600 text-white font-medium rounded-lg hover:bg-blue-700 focus:outline-none focus:ring-2 focus:ring-blue-500 focus:ring-offset-2 transition-colors` to `btn btn-primary`). The semantic color system (`primary`, `secondary`, `accent`, `base-100/200/300`, `error`, `success`, `warning`, `info`) enables instant theme switching and future customization. Phase 3's conversation interface will heavily benefit from DaisyUI's `chat`, `steps`, `drawer`, and form components.

**Primary recommendation:** Install DaisyUI v5 as devDependency, add `@plugin "daisyui"` with corporate/business themes, then migrate all existing components file-by-file to DaisyUI classes while preserving all functionality.

## Architecture Patterns

### CSS Configuration Pattern
```css
/* src/index.css */
@import "tailwindcss";
@plugin "daisyui" {
  themes: corporate --default, business --prefersdark;
}
```

This is the ONLY configuration change needed. No `tailwind.config.js`, no `postcss.config.js`, no `vite.config.ts` changes.

### Theme Application Pattern

Set `data-theme` on the `<html>` element in `index.html`:
```html
<html lang="en" data-theme="corporate">
```

DaisyUI automatically switches to `business` theme when `prefers-color-scheme: dark` is detected (via `--prefersdark` flag). For manual switching, use `document.documentElement.setAttribute('data-theme', themeName)`.

### Anti-Patterns to Avoid
- **Mixing hard-coded colors with semantic colors:** Once migrated, ALL colors should use DaisyUI semantic names (primary, base-100, etc.) not Tailwind color values (blue-600, gray-50). This ensures themes work correctly.
- **Wrapping DaisyUI in React component abstractions prematurely:** DaisyUI classes are already the abstraction. Don't create `<Button variant="primary">` wrappers until a clear need arises (Phase 3+ may warrant it).
- **Using `<dialog>` for blocking modals:** The native `<dialog>` allows ESC dismissal. For the ConsentModal which MUST NOT be dismissible, use the `modal-open` CSS class approach with React state.
- **Adding redundant utility classes:** DaisyUI's `btn` already includes padding, border-radius, transitions, focus states. Don't add `px-6 py-2.5 rounded-lg transition-colors` on top of `btn`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Button styles | Custom button utility classes | `btn btn-primary`, `btn btn-outline` | 8 colors, 5 sizes, 6 style variants, disabled/loading states built-in |
| Modal overlay/centering | Fixed position + backdrop div | `modal` + `modal-box` + `modal-open` | Handles backdrop, centering, responsive positioning, scroll locking |
| Alert/notification styling | Custom colored divs with borders | `alert alert-error`, `alert-soft` | Semantic colors, icon support, responsive layout |
| Form input styling | Custom border/focus/ring utilities | `input`, `select`, `textarea` | Consistent sizing, focus states, color variants, validation integration |
| Loading spinners | CSS animations or SVG spinners | `loading loading-spinner` | 6 animation types, 5 sizes, theme-aware colors |
| Dark mode support | Manual `dark:` prefix on every element | DaisyUI themes with `--prefersdark` | Single config line handles entire app theme |
| Link underline styles | Manual underline + hover utilities | `link link-primary` | Consistent with theme, hover variant available |
| Card containers | `bg-white rounded-lg shadow p-6` | `card bg-base-100 shadow-sm` + `card-body` | Semantic structure, size variants, theme-aware |
| Theme switching | Custom context + CSS variables | `theme-controller` + `data-theme` | CSS-only mechanism, localStorage for persistence |
| Form field grouping | Custom label + spacing | `fieldset` + `fieldset-legend` + `label` | Semantic HTML, consistent spacing, accessible |

**Key insight:** Every hand-rolled UI pattern in the current codebase has a DaisyUI equivalent that is theme-aware, accessible, responsive, and dramatically less verbose. The migration is a net reduction of CSS complexity.

## Common Pitfalls

### Pitfall 1: Hard-Coded Colors Breaking Theme Switching
**What goes wrong:** Using Tailwind color utilities (`bg-blue-600`, `text-gray-900`) alongside DaisyUI components. When user switches theme, DaisyUI components change but hard-coded colors stay fixed.
**Why it happens:** Partial migration -- some elements converted, others left with Tailwind color classes.
**How to avoid:** Migrate ALL color references in a component at once. Use `bg-primary`, `text-base-content`, `bg-base-100`, etc. Search for remaining `gray-`, `blue-`, `red-`, `green-` references after migration.
**Warning signs:** Visual inconsistency when switching themes; some elements change color, others don't.

### Pitfall 2: Redundant Utility Classes on DaisyUI Components
**What goes wrong:** Adding utility classes that DaisyUI already includes, causing specificity conflicts or unexpected sizing.
**Why it happens:** Developers used to writing Tailwind utilities don't realize DaisyUI's component class already handles padding, border-radius, focus states, etc.
**How to avoid:** When adding a DaisyUI class, REMOVE the equivalent utilities. `btn` replaces: padding, border-radius, font-weight, focus ring, hover transition, cursor. Only add layout utilities (`w-full`, `flex-1`).
**Warning signs:** Buttons/inputs look "off" compared to DaisyUI examples; double padding or wrong border radius.

### Pitfall 3: ConsentModal Dismissibility
**What goes wrong:** Using `<dialog>` element for the consent modal, which allows ESC key or backdrop click to dismiss it. POPIA consent MUST be explicit -- the modal must block until accepted.
**Why it happens:** DaisyUI docs recommend `<dialog>` as the primary modal approach.
**How to avoid:** Use `modal-open` class controlled by React state for the ConsentModal. The `<dialog>` approach is fine for other, dismissible modals in future phases.
**Warning signs:** User can bypass consent by pressing ESC.

### Pitfall 4: Missing `data-theme` on Root Element
**What goes wrong:** Themes don't apply; components render with default/unstyled appearance.
**Why it happens:** Forgot to add `data-theme` attribute to `<html>` element, or DaisyUI config doesn't specify themes.
**How to avoid:** Add `data-theme="corporate"` to `<html>` in `index.html`. Verify the `@plugin "daisyui"` config includes the themes you're using.
**Warning signs:** Components look like they have no styling, or all components are one color scheme regardless of theme setting.

### Pitfall 5: Clerk UI Components Conflict
**What goes wrong:** DaisyUI's base styles might affect Clerk's `<SignInButton>`, `<UserButton>` rendered UI.
**Why it happens:** DaisyUI applies global base styles via the `@plugin` directive.
**How to avoid:** Clerk components render in their own portal/iframe and should be unaffected. The `<button>` elements INSIDE `<SignInButton>` and `<SignUpButton>` (custom trigger buttons) should use DaisyUI classes since those are our elements. `<UserButton>` is entirely Clerk-managed and should remain untouched.
**Warning signs:** Clerk modal/popover looks unstyled or has conflicting styles.

### Pitfall 6: Forgetting `select` Default Option Pattern
**What goes wrong:** DaisyUI `select` doesn't show placeholder text.
**Why it happens:** DaisyUI v5 changed how default values work with selects.
**How to avoid:** Use `defaultValue` prop in React with a disabled first option: `<option disabled selected>Pick one</option>`.
**Warning signs:** Select shows first real option instead of placeholder.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `tailwind.config.js` with `require('daisyui')` | `@plugin "daisyui"` in CSS file | DaisyUI v5 (Jan 2025) | No JS config files needed |
| `form-control` class for form groups | `fieldset` + `fieldset-legend` | DaisyUI v5 | Semantic HTML, better accessibility |
| `btm-nav` for bottom navigation | `dock` component | DaisyUI v5 | New component with 5 size options |
| `darkTheme` config in JS | `--prefersdark` flag in CSS | DaisyUI v5 | Theme config entirely in CSS |
| Short CSS variables (`--p`, `--s`, `--a`) | Full names (`--color-primary`, `--color-secondary`) | DaisyUI v5 | Readable, self-documenting |
| `input-bordered` class | Borders on by default, use `input-ghost` for borderless | DaisyUI v5 | Less classes needed |
| `card-compact` | `card-sm` | DaisyUI v5 | Consistent with size naming scheme |
| Auto-generated `-content` colors | Manual content color definition | DaisyUI v5 | More control, less magic |

**Deprecated/outdated:**
- `form-control` class: Removed, use `fieldset` component
- `btn-group` class: Removed, use `join` component
- `input-group` class: Removed, use `join` component
- `input-bordered`, `select-bordered`, `textarea-bordered`: No longer needed (borders are default)
- `card-bordered`: Renamed to `card-border`

## Open Questions

1. **Corporate vs Nord theme for light mode**
   - What we know: `corporate` is explicitly business-oriented. `nord` has a clean, professional feel. Both work well for legal/professional contexts.
   - What's unclear: Which better matches "trustworthy South African legal platform" aesthetic.
   - Recommendation: Use `corporate` as default (it is literally named for business use). Easy to switch later via one-line CSS change.

2. **Custom theme refinement timing**
   - What we know: Built-in themes work immediately. Custom themes can override specific colors.
   - What's unclear: Whether the project will need SA-specific colors (e.g., green/gold national colors) beyond what built-in themes provide.
   - Recommendation: Start with built-in themes. Defer custom theme creation until stakeholder feedback indicates specific branding needs.

3. **Clerk component style isolation**
   - What we know: Clerk renders auth UI in portals/iframes. Our custom trigger buttons (inside `<SignInButton>`, `<SignUpButton>`) are standard HTML we control.
   - What's unclear: Whether DaisyUI base styles leak into Clerk's rendered modal in any edge cases.
   - Recommendation: Test during implementation. If conflict occurs, Clerk's `appearance` prop can override styles.
